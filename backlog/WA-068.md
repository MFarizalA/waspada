# WA-068 · Single-file dashboard build so the live URL gives judges a working app

## Context

The live dashboard must be submittable — judges open the URL. The `*.fcapp.run`
test domain force-downloads `text/html` (WA-018 quirk, platform-level, no app
fix possible). But the whole dashboard is 242KB: one JS (197KB) + one CSS (28KB).

If we inline ALL assets into a single self-contained `index.html`, then even
when the browser downloads the file, **double-clicking it opens the complete
working app** — styled, interactive, calling the live API via CORS (already
verified: FC echoes `Access-Control-Allow-Origin` and the app allows
credentials; the seeded judge account is `analyst@waspada.demo` / `waspada123`).

## Goal

Add a single-file build mode to the dashboard: `index.html` with ALL JS/CSS
inlined (no external `/assets/*` references). The deployed image serves this
single-file version.

## Files / interfaces

- `dashboard/vite.config.ts` — add `vite-plugin-singlefile` (add to
  devDependencies; it's the standard tool for this) with
  `inlineDynamicImports: true` in the build config
- `dashboard/package.json` — new dep
- `Dockerfile` (root) — verify the image build still works with the plugin
  (npm ci will fetch it; no other change needed)
- `api/main.py` `/` route — no change needed (HTMLResponse serves whatever
  index.html exists), BUT verify the StaticFiles `/assets` mount doesn't 404
  anything the single-file build still references

## Acceptance

- `cd dashboard && npm run build` produces a `dist/index.html` that contains
  ALL JS + CSS inline (no `<script src=` or `<link rel="stylesheet" href=`
  pointing at `/assets/`).
- Opening `dist/index.html` directly from disk (file://) renders the full
  styled dashboard and can log in against the live prod API
  (`https://waspadaprod-api-vouqzqqkiu.ap-southeast-1.fcapp.run`) — verify
  login works from the file:// origin (CORS must accept it; if the API
  rejects file:// origins, report it — that's a one-line CORS config fix for
  Stefanie/Bimo, not something you hack around).
- `npm run dev` (local vite dev) still works unchanged.
- CI image build passes (push branch, watch build-image.yml run).
- dist size sanity: single file < 2MB (we're at 242KB + inline overhead —
  anything past ~1MB means something double-bundled).

## Out of scope

- Custom domain (WA-067, deferred).
- Any API/backend changes (CORS fix, if needed, goes to Bimo as a separate
  ticket — flag it in your completion note).
- Changing dashboard features/content.

## Notes

- Keep the multi-asset build possible (env flag or a separate config) if it's
  trivial — but single-file becomes the DEFAULT build since that's what ships.
  If the flag adds complexity, just switch the default and note it.
- vite-plugin-singlefile is MIT, ~1KB, standard. If it misbehaves with our
  setup, the fallback is a tiny post-build node script that inlines the two
  asset files into index.html — 20 lines, your call.
