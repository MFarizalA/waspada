# WA-069 · End-to-end backend verification: agents, debate, pipeline against live prod

## Context

Owner directive (2026-07-18): "Make sure all backend system, agents, and data
pipeline works properly." Frontend focus comes later. The deployment is live
(`https://waspadaprod-api-vouqzqqkiu.ap-southeast-1.fcapp.run`, health ok) and
425 tests are green on develop — but green unit tests ≠ proven end-to-end
behavior against the live system. This ticket is the systematic proof pass.

## Goal

Produce a verification report proving each backend subsystem works — locally
against real code paths AND against the live prod deployment where applicable.

## What to verify (numbered, each gets a verdict + evidence)

1. **Pipeline orchestration** — run the full Collections pipeline locally
   (mock brain, stubbed fetch — the standard offline path): ingest → features →
   scoring → ranking → all agents execute without error. Evidence: run output
   showing every agent's AgentResult with sane statuses.
2. **Debate protocol end-to-end** — execute a live debate via the prod API:
   `GET /api/run?brain=mock` (mock first — cheap). Verify: rounds progress
   (challenge → defense → ruling), resolutions are valid (open/concede/upheld/
   escalate), `agent_dialogue` is populated, portfolio Expected Loss appears in
   the output (WA-024 regression check). If mock passes and time allows, ONE
   live Qwen run (`brain=qwen`, K=1-2) to prove the real brain works in prod —
   but check with Stefanie first if it burns paid tokens.
3. **Data pipeline reads** — verify the Data Analyst / Data Engineer agents
   read from the real surface: DuckDB views over OSS parquet (WA-047 made this
   the only path). No dlt remnants, no silent local-file fallback. Evidence:
   the agent's SQL/tool calls against the mart layer succeed and return
   non-empty results.
4. **Auth flow against prod** — register/login/me against the live API (use a
   throwaway account, not the seeded analyst). JWT issued, protected route
   accepts it, bad token rejected. The seeded `analyst@waspada.demo` login
   must also work (judges use it).
5. **Risk decision matrix (WA-032)** — load the human-editable JSON, run a
   score through it, verify band assignment matches the matrix (including a
   boundary value at an exact band edge — WA-051 regression check).
6. **Actuary introspection (WA-050)** — verify the Actuary's defense cites the
   model's REAL fitted coefficients: fit on synthetic data with known
   coefficients, run the defense, confirm cited values match fitted values.
7. **MCP evidence calls** — the Risk Auditor's claims cite MCP-pulled evidence
   (`portfolio_stats`, `lookup_account`). Verify the MCP server responds and
   the cited numbers match what the tools return.

## Method

- Local runs use the repo venv: `.venv/Scripts/python.exe -m pytest` for the
  suite-level re-confirmation, plus direct script runs for the orchestrated
  paths (scripts/ or `python -m waspada...` — whatever the repo already has;
  write a small throwaway verification script under `scripts/` if needed and
  commit it — reusable verification tooling is welcome).
- Prod calls: plain `curl` against the fcapp.run URL. All `/api/*` routes
  serve inline JSON.
- Do NOT touch production data (RDS/OSS) beyond what the API endpoints
  themselves do. Read-only posture.
- If anything is broken: do NOT fix it in this ticket. Document it precisely
  (repro, expected vs actual, suspected layer) — fixes get their own tickets.

## Acceptance (your report)

`backlog/WA-069-report.md` with, per numbered item:
- **Verdict:** ✅ works / ⚠️ works-with-caveat / ❌ broken
- **Evidence:** command run + key output lines
- **Notes:** anything the owner should know (latency, token cost, caveats)

Plus a one-paragraph overall summary: is the backend demo-ready, and what's
the single biggest risk if not.
