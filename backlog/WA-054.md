# WA-054 · Concurrency is unmodeled

## The gap

Function Compute runs with `instance_concurrency = 10` (`deploy/iac/main.tf`), but
the whole run is written as if it were the only one in flight:

- Two concurrent runs over the same book would **both** open disputes on the same
  `loan_id`s and debate them independently — duplicate LLM spend, and two
  potentially *different* rulings on the same account.
- The dispute-memory index is a read-modify-write of a single object
  (`LocalFileMemory`, `dispute_memory.py:114-159`), so concurrent invocations
  silently drop each other's writes. **WA-046 fixes this half** by moving to
  per-loan objects.

## The fix — pick one and state it

- **(a)** Take a `lane + as_of` **run lock** (an OSS object with a TTL, or an RDS
  advisory lock — RDS is already there for auth). Concurrent runs on the same book
  block or no-op.
- **(b)** Explicitly **declare runs serialized** and set `instance_concurrency = 1`,
  documenting that horizontal scale is per-lane, not per-run.

Either is acceptable. What is **not** acceptable is leaving it undefined while the
IaC advertises concurrency 10.

## Acceptance

- The chosen policy is documented in HACKATHON.md § architecture and enforced in
  `deploy/iac/main.tf`.
- Two simultaneous runs on the same `lane + as_of` do not double-spend the debate
  budget or produce conflicting resolutions.

## Depends on

**WA-046** (per-loan dispute objects) fixes the memory-corruption half.
