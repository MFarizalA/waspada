# WA-088 · Scheduled batch refresh — fresh data via an FC Time Trigger

**Candidates:** Claude or backend/infra. Makes "fresh data" real for production instead of a
one-time manual upload.

## Problem
The society reads the OSS book **on-demand per `/api/run`**, but the OSS object is a **one-time
upload → it goes stale**. Collections is a **daily batch** discipline (score the portfolio each
morning). Without a scheduled producer, every run scores a frozen snapshot.

## Goal
A **scheduled FC Time Trigger** (cron) that periodically refreshes the OSS loan book — and
optionally pre-computes the dashboard payload — so the society always scores current data.

## Approach
- Add an FC **Time Trigger** (e.g. `CRON_TZ=Asia/Jakarta 0 0 6 * * *` = 6am daily) on a small
  refresh function (or a dedicated FC function).
- The refresh job: pull the latest loan book from source → transform to RawLoans → write the OSS
  Raw object. **Prefer the dlt pipeline (WA-083)** — incremental cursors + `merge` dedup load only
  new/changed loans; `_dlt_loads` lineage stamps freshness.
- Optional: a second scheduled step (or the same job) runs the society + writes the pre-computed
  `DashboardPayload` to OSS Mart, so the dashboard shows a fresh cached run instantly.
- Idempotent + fail-safe: a failed refresh leaves the last-good object in place; alert via SLS.

## Ties to other work
- **WA-083 (dlt):** the refresh job's load step *is* the dlt incremental pipeline.
- **WA-082 (model versioning):** the same cadence can retrain + republish the PD model to OSS.
- **WA-087 (secrets):** the scheduled function uses the RAM role (no static keys) for OSS.

## Acceptance
- An FC time trigger runs on a documented cadence and refreshes the OSS Raw object.
- The Data Engineer's freshness gate sees a recent `last-modified` / `_dlt_loads` timestamp.
- Failure leaves the previous book intact; the failure is visible (SLS/audit).

## Out of scope
- The initial one-time data land (WA-078).
- Real-time/streaming ingest (batch cadence is the collections pattern).

## Notes
Demo doesn't need this (a one-time upload suffices); it's the **real-production** freshness
mechanism. Sequence after WA-078 (data lands) and ideally WA-083 (dlt). Touches IaC (the time
trigger) + a refresh entrypoint — coordinate the apply with the deploy lane.

## OSS object-key convention (owner, 2026-07-19) — date-partitioned, NOT a flat object
```
bucket: waspada-prod-raw
key:    loans/dt=<YYYYMMDD>/loans.parquet         e.g. loans/dt=20260718/loans.parquet
```
- `<YYYYMMDD>` = ISO date of the batch (year-month-day). Lexicographic order == chronological,
  so the reader's "latest partition" is a plain `max()` on the partition names.
- **Writers** (the WA-078 loader + this scheduled refresh) land a **new `dt=` partition** each run.
- **Reader** (`waspada/data/oss.py`) must resolve the **LATEST `dt=` partition**, not a fixed
  `OSS_KEY` — this is exactly the **WA-047** "OSS_KEY → OSS_PREFIX + latest-partition resolver" item.
- **Current state:** the landed 50k-row real dataset sits at the **flat `loans.parquet` (root)**;
  re-land it under the partitioned path when the data work resumes (verified 2026-07-19: valid
  RawLoans, real Lending Club vocabulary — Fully Paid / Charged Off / Late / In Grace Period).
