## Goal

The submission requires three documentation artifacts that must exist and be
correct: an architecture diagram, a proof-of-Alibaba-Cloud code link, and the
text description. Plus one final docs-consistency sweep.

## What to produce

1. **Architecture diagram** (`docs/architecture.png` or `.svg`, linked from
   README). Must clearly show: Qwen Cloud (DashScope) ↔ backend agents
   (FastAPI on FC), OSS (Raw/Staging/Mart buckets), RDS MySQL (auth), SLS
   (audit), ACR → FC deploy flow, and the dashboard. One diagram a judge
   understands in 30 seconds. Excalidraw/drawio/SVG — author's choice, but
   commit the source file too.
2. **Proof of Alibaba Cloud deployment.** A link (for the submission form)
   to a code file demonstrating Alibaba Cloud services/APIs. Best candidate:
   `deploy/iac/main.tf` (OSS + ACR + FC + SLS + RDS resources) — verify the
   file tells the story standalone. Backup: `waspada/data/oss.py` (OSS SDK
   usage) or the SLS audit wrapper. Decide, and note the exact GitHub URL
   (must work once repo is public — use the main-branch permalink).
3. **Submission text description** (`docs/submission-description.md`). The
   features/functionality text for the submission form: problem, the agent
   society design (industry names), Qwen Cloud usage per agent tier, Alibaba
   Cloud architecture, the debate protocol, deployment proof. ~300-500 words,
   written for judges, no internal codenames.
4. **Consistency sweep.** Fix the WA-064-flagged drift: README/HACKATHON
   still describe dlt as live (it's removed — WA-047), Data Engineer
   docstring mentions dlt. Docs must match reality — judges read then poke.
