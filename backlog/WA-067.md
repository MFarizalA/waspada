# WA-067 · Custom domain for the FC deployment (kill the forced-download quirk)

**Do not dispatch until owner says go — post-hackathon ticket.**

## Context

The deployed dashboard at the `*.fcapp.run` URL downloads as an attachment
instead of rendering in the browser. Root-caused 2026-07-18 by fingerprinting
live response headers:

- `Content-Disposition: attachment` is injected by **Function Compute's
  default `*.fcapp.run` test domain** as an anti-abuse / anti-phishing
  measure on the shared domain. It only hits `text/html` responses (the `/`
  dashboard route); all `/api/*` JSON routes serve inline normally.
- The app is NOT at fault: absence of `ETag`/`Last-Modified`/`accept-ranges`
  in the live response proves the fixed `HTMLResponse` path is serving, and
  no app response object can remove a header the platform adds *after* the
  response leaves the container.

The deployed API is fully functional — this is a UX/polish issue on the
dashboard URL only, not a correctness bug.

## Goal

Bind a custom domain to the FC function so the dashboard renders inline in
the browser. Custom domains on FC do not get the forced-download treatment.

## Files / interfaces

- `deploy/iac/main.tf` — add `alicloud_fcv3_custom_domain` + route to
  `alicloud_fcv3_function.api`
- DNS (owner's registrar) — CNAME record → the FC endpoint
- `README.md` / `HACKATHON.md` — update the deploy URL once cut over

## Acceptance

- Dashboard at the custom domain renders inline in the browser (no
  `Content-Disposition: attachment` header in the response — verify with
  `curl -sI`).
- `/api/health` on the custom domain returns `{"status":"ok"}`.
- Old `*.fcapp.run` URL continues to work (FC keeps it unless explicitly
  removed).
- `tofu plan` shows only the custom-domain resources added; no recreation
  of the function/trigger.

## Prerequisites (owner)

1. A domain or subdomain we control (purchase or repurpose an existing one).
2. DNS access to add the CNAME.
3. If HTTPS is wanted (it is): cert via Alibaba SSL or Let's Encrypt —
   decide at implementation time.

## Out of scope

- Changing any app code (the HTMLResponse fix is correct and stays).
- Removing the fcapp.run URL.
- CDN / WAF in front of the domain (separate ticket if ever needed).

## Notes

The demo workaround (documented in README/HACKATHON): dashboard is served
locally against the deployed API; all `/api/*` endpoints on the fcapp.run
URL serve JSON inline without issue.
