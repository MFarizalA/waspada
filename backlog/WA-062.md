# WA-062 · Prepare auth database — schema init + connection verification against live RDS

## The gap

The auth system (WA-028) has schema definitions in `api/db.py` but the tables
have never been created on the live RDS MySQL instance. The `DATABASE_URL`
env var is empty in `.env`, so the app can't connect in production.

This ticket:
1. Wires `DATABASE_URL` in `.env` to the live RDS MySQL instance
2. Creates the auth tables (`users`, `reset_tokens`) on the live instance
3. Verifies the full auth flow works against the real database (not SQLite)

## Schema (from `api/db.py:47-65`)

```sql
CREATE TABLE IF NOT EXISTS users (
    email       TEXT PRIMARY KEY,
    password    TEXT NOT NULL,
    created_at  TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS reset_tokens (
    token       TEXT PRIMARY KEY,
    email       TEXT NOT NULL,
    expires_at  TEXT NOT NULL,
    used        INTEGER NOT NULL DEFAULT 0,
    FOREIGN KEY (email) REFERENCES users(email)
);
```

**MySQL notes:** `TEXT` as PRIMARY KEY needs a length in MySQL.
`api/db.py` already handles this — the `_MySQLDb._create` method renders
`email VARCHAR(255) PRIMARY KEY` for MySQL. Use `init_db()` to create the
tables; don't hand-write SQL.

## Files / interfaces

1. **`.env`** — set `DATABASE_URL`:
   ```
   DATABASE_URL=mysql+pymysql://waspada:<rds_password>@rm-gs567k8z72rc11wds.mysql.singapore.rds.aliyuncs.com:3306/waspada
   ```
   The `rds_password` is in `deploy/iac/secrets.tfvars` under `rds_password`.

2. **`api/db.py`** — no changes needed. `init_db()` already:
   - Detects MySQL vs SQLite from `DATABASE_URL`
   - Creates tables if they don't exist
   - Returns the right adapter

3. **New script: `scripts/init-auth-db.py`** — a one-shot script that:
   - Calls `init_db()` with the production `DATABASE_URL`
   - Prints the tables created and their schemas
   - Runs a smoke test: register → login → forgot-password → reset
   - Cleans up the test user
   - Prints connection latency (useful for FC cold-start estimates)

4. **Verify auth against live RDS:**
   - `POST /api/auth/register` creates a user in the `users` table
   - `POST /api/auth/login` returns a JWT
   - `POST /api/auth/forgot-password` creates a row in `reset_tokens`
   - `POST /api/auth/reset-password` marks the token used
