# WA-038 · Data agents per lane (Data Engineer / Data Analyst / OSS source)

## Goal

Point the two Tier-2 data agents and the OSS source at the right lane. Their
plumbing is already lane-aware (`fetch(lane=lane, ...)` —
`data_engineer.py:98-108`); their **bodies are collections-wired**.

## What's hardwired today

- **Data Analyst** imports and calls `build_features` from
  `features.collections` (`data_analyst.py:48,111`); its loop prompts reference
  `payment_ratio`/`outstanding_ratio`/`total_paid` (`data_analyst.py:221-234`).
- **Data Engineer** anomaly checks are collections-schema-specific
  (`term NOT IN (36,60)`, `dti>100`, `amount<0` … `data_engineer.py:373-378`)
  and the deterministic gate validates against `RawLoans`.
- **OSS client** selects the same `_RAW_LOANS_COLUMNS`/object regardless of lane
  (`data/oss.py:115`).

## Files / interfaces

- `data_analyst.py`: select the feature builder + contract by `context.lane`
  (collections → `features.collections.build_features`; origination →
  `features.origination.build_features` from WA-035); lane-appropriate prompt
  table/column descriptions.
- `data_engineer.py`: lane-appropriate schema gate (`RawLoans` vs
  `RawApplications`) + anomaly check set (e.g. `requested_rate` bounds,
  `amount/annual_income` sanity) — keep the check set data-driven so the LLM
  loop and the deterministic fallback share it.
- `data/oss.py`: per-lane object key + column projection (`OSS_KEY` stays
  collections; add `OSS_ORIGINATION_KEY` or a lane-suffixed convention).

## Acceptance

- Both agents run the origination lane offline on a synthetic
  `RawApplications` snapshot (stubbable `fetch`, same `register_tool` pattern).
- Collections behavior unchanged (existing agent tests green).
- The Data Analyst regression guard holds per lane: the frame equals the
  deterministic recipe's output (same rule as WA-030).
