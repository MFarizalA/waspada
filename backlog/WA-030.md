# WA-030 · AI Data Engineer + AI Data Analyst agents

## Goal

Upgrade the two deterministic pipeline agents into AI agents powered by Qwen
function calling + the DuckDB query layer (WA-029). The Data Engineer validates
and profiles the data; the Data Analyst explores features and surfaces insights.
Both use Qwen to decide WHICH queries to run — genuine AI-powered analytics,
not static Python.

## The two agents

### Data Engineer Agent (replaces IngestAgent)
- Role: "Validate, profile, and quality-check the data snapshot"
- Brain: qwen3.6-flash (cheap, fast — same tier as Risk Auditor)
- Tools (via function calling):
  - `validate_schema()` — check the loaded table matches RawLoans contract
  - `row_count()` — total rows
  - `null_rates()` — per-column null rates
  - `freshness_check()` — is the snapshot stale?
  - `profile_column(name)` — distribution, min/max/mean for a column
  - `detect_anomalies()` — flag outliers (e.g., dti > 100, rate < 0)
- Behavior: LLM receives the table metadata + decides which checks to run.
  It reasons: "This is a collections snapshot — I should check null rates on
  payment_ratio and dti, verify the grade distribution looks right, and flag
  any loans with impossible values."
- Output: a data quality report (passed to the Data Analyst + surfaced in audit)
- Fallback: if LLM unavailable, run the deterministic checks (current behavior)

### Data Analyst Agent (replaces AnalyticsAgent)
- Role: "Build features, explore the data, surface insights for the risk model"
- Brain: qwen3.7-plus (mid tier — needs more reasoning for feature engineering)
- Tools (via function calling):
  - `query(sql)` — run a DuckDB SQL query (read-only, sandboxed)
  - `correlation(col_a, col_b)` — Pearson correlation
  - `distribution(col)` — histogram bins
  - `segment_stats(group_col, metric_col)` — group-by aggregation
  - `build_feature(name, sql)` — define a new feature column via SQL
- Behavior: LLM receives the data quality report + schema + decides:
  "I need payment_ratio (total_paid/amount), outstanding_ratio, loan_age in
  months. Let me also check the DTI distribution by grade — if grade E has
  abnormally high DTI, that's worth flagging for the model."
- Output: the FeatureFrame (same shape as today) + an insights summary
- Fallback: if LLM unavailable, run the deterministic feature build

## Function-calling loop

Both agents use Qwen's native function calling (the same mechanism as the Risk
Auditor). The loop is:

1. Send system prompt + table metadata + available tools (JSON schema)
2. Qwen returns `tool_calls` → execute → send results back
3. Qwen may call another tool (multi-hop) → execute → send results back
4. Qwen returns final JSON (quality report / feature spec)
5. Validate JSON → safe fallback on parse failure

This is the multi-hop tool-calling loop the rubric rewards (Criterion 1:
"sophisticated use of QwenCloud APIs").
