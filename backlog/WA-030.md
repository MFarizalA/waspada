# WA-030 ¬∑ Data Analyst agent (qwen3.7-plus function-calling + DuckDB SQL)

## Goal

Promote `analytics` into a Tier-2 Data Analyst agent: a `qwen3.7-plus`
function-calling loop over DuckDB SQL that *explores* the book for the aggregates
the debate later cites, while still emitting the frozen `FeatureFrame`. The fixed
feature recipe (`build_features`) remains the deterministic core; the aggregates
the agent computes back the MCP evidence base (¬ß Analytics), so what an agent
cites downstream is what this step computed.

## THE LOAD-BEARING DECISION (do not get this wrong)

The LLM chooses **which explorations to run**. The LLM does **not** compute the
`FeatureFrame` numbers. The `FeatureFrame` is still produced by `build_features()`
so the output is byte-for-byte reproducible ‚Äî that is the "regression guard"
acceptance below. The agent therefore has **two** outputs:

1. `feature_frame` handle ‚Äî identical to what `AnalyticsAgent` produces today
   (via `build_features`). Non-negotiable; a regression test pins it. **Handle
   name stays `feature_frame`** ‚Äî `risk_model` and the whole chain depend on it.
2. `analyst_aggregates` handle (new) ‚Äî the correlations/distributions/group-bys
   the agent chose to run, stashed on the context to back the MCP evidence base.
   This is the new value the LLM adds. It must never feed back into the feature
   numbers.

Loop failure / unparsable brain / brain unavailable ‚Üí fall back to plain
`build_features` + empty aggregates. Pipeline never crashes; contract always
satisfied.

## Tools to register (read-only DuckDB SQL; `register_tool` pattern)

Each takes the `Lakehouse` (built over RawLoans+FeatureFrame in `run()`) and
returns a JSON-serialisable dict. **SELECT-only** ‚Äî reject any non-SELECT SQL
(`.strip().lower().startswith("select")` + reject `;`-chained statements; brain
output is untrusted text).

- `query`         ‚Äî a bounded `SELECT` the LLM composes (cap rows, e.g. `LIMIT 200`).
- `correlation`   ‚Äî Pearson corr between two numeric columns (`corr(a,b)`).
- `distribution`  ‚Äî quantiles/min/max/mean/histogram buckets for one column.
- `build_feature` ‚Äî return an aggregate the debate cites (e.g. segment medians),
  **NOT** a mutation of the FeatureFrame. Read-only aggregate computation.

## Order of operations in `run()`

1. Consume the predecessor `raw_loans` handle (same predecessor-lookup pattern
   `AnalyticsAgent.run` uses).
2. Build `frame = build_features(raw, as_of)` + `assert_no_nulls`
   **unconditionally, first** ‚Äî guarantees the contract regardless of the loop.
3. Build a `Lakehouse` over `raw` (and optionally the frame) for the tools.
4. Run the exploration loop ‚Üí collect `aggregates`.
5. Publish `context.data_handles["feature_frame"] = frame` and
   `context.data_handles["analyst_aggregates"] = aggregates`.
6. Return `AgentResult(status=OK, artifact_ref="feature_frame", ...)`.

## Acceptance

- Real multi-hop tool loop (Qwen decides the next query from prior results),
  logged step-by-step in `self.step(...)` (the audit trail is graded ‚Äî the
  multi-hop run must be visible in the log).
- Any tool/parse failure ‚Üí falls back to the deterministic feature recipe;
  pipeline never crashes, FeatureFrame contract always satisfied.
- **Regression guard:** `DataAnalystAgent`'s `feature_frame` is *identical*
  (field-for-field Arrow compare) to `build_features` output on the offline
  snapshot ‚Äî not "close," identical.
- Offline tests green on MockLLM.

## Tests (new: `tests/test_data_analyst.py`) ‚Äî green offline on MockLLM
