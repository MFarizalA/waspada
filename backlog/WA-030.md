# WA-030 · Data Analyst agent (qwen3.7-plus function-calling + DuckDB SQL)

## Goal

Promote `analytics` into a Tier-2 Data Analyst agent: a qwen3.7-plus
function-calling loop over DuckDB SQL that builds features and explores the book
for the aggregates the debate later cites. The fixed feature recipe remains the
deterministic fallback; the aggregates the agent computes back the MCP evidence
base (§ Analytics), so what an agent cites downstream is what this step computed.

## Files / interfaces

- `waspada/agents/data_analyst.py` (new) — the agent + its function-calling loop.
- DuckDB query tools via `register_tool`: `query`, `correlation`,
  `distribution`, `build_feature` (SQL pushed down to DuckDB over the Parquet).
- `waspada/agents/orchestrator.py` — swap `analytics` → `data_analyst` in the
  step order.
- Must still emit the frozen FeatureFrame shape + the portfolio stats the MCP
  tools serve (WA-015 depends on these aggregates existing).

## Acceptance

- Real multi-hop tool loop (Qwen decides the next query from prior results),
  logged step-by-step.
- Any tool/parse failure → falls back to the deterministic feature recipe;
  pipeline never crashes, FeatureFrame contract always satisfied.
- Feature output matches the pre-agent pipeline on the offline snapshot
  (regression guard — the agent explores, but the numbers are reproducible).
- Offline tests green on MockLLM.

## Out of scope

- Data validation / quality checks (WA-029).
- New feature *definitions* beyond what the debate needs (keep scope tight).
- The efficiency benchmark (WA-017) — unaffected by this swap.

## Depends on

- WA-029 (DuckDB query layer + dlt ingestion must exist for the agent to call).
