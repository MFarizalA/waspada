# WA-095 · Human-defined parameter matrix (governance control surface)

**Candidates:** Claude or full-stack. Deepens the "human + agent society" story: today the human's
only in-loop moment is approving/rejecting one dispute at the gate. This lets the human **define the
rules the whole society plays by**, before the run — the governance layer a real risk/collections
officer owns.

## Why (grounded — these knobs already exist, just hardcoded)
The society's behaviour is governed by constants baked into code / env, invisible and unowned by the
human:

- **Band → action matrix** — `insight/ranking.py::ACTION_BY_BAND` (`Very High → call`,
  `High/Medium → watch`, `Low → auto-cure`). A literal parameter *matrix*, hardcoded.
- **Risk band thresholds** — `band_edges` (absolute PD cutpoints; WA-051/WA-093).
- **Dispute admissibility gap** — `risk_auditor.DISPUTE_GAP` (=2): how far the Skeptic must diverge
  to open a dispute.
- **Arbiter escalation threshold** — `arbiter.ARBITER_CONFIDENCE_THRESHOLD` (=0.6): below this the
  arbiter punts to the human.
- **Audit budget / work-list cap** — `audit_k` (=8), `top_n`.
- **Cohort alert thresholds** — `DEFAULT_NPL_THRESHOLD` (0.20), `DEFAULT_VINTAGE_THRESHOLD` (0.15).

These are **policy**, not model internals — the human should set them. It's also a governance/audit
requirement: *"which policy matrix governed this run?"* must be answerable.

## Design
- **A "Parameter Matrix" panel** in the dashboard where the analyst defines policy before a run:
  - the **band × action grid** (editable matrix — the centrepiece),
  - the **risk-band PD thresholds** (band_edges),
  - **dispute gap**, **arbiter confidence threshold**, **audit K**, **top-N**,
  - **NPL / vintage alert thresholds**.
- **Thread it through the run.** The plumbing largely exists — the `Orchestrator` already takes
  `policy` / `top_n` / `audit_k`, and `ranking.rank(action_by_band=…)` already accepts a matrix
  override. Wire the human's matrix from the UI → `/api/run` (+ `/api/run/stream`) → these params, so
  the work-list, disputes, and alerts all honour it.
- **Provenance / audit.** Every run records **which matrix** governed it (a hash/version + the values)
  in the step log / audit trail — so a decision is always traceable to the policy that produced it.
  Pairs with WA-082/WA-093 (which model + which policy governed run X).
- **Persistence.** Save named matrices per analyst/session (a "conservative" vs "aggressive" preset),
  recallable. Ship 1–2 sensible presets; the current constants are the **default** preset.
- **Guardrails.** Validate ranges (thresholds in [0,1], K/top-N positive ints, actions ∈ the known
  set); reject nonsensical matrices with a clear message; bound values so a human can't wedge the run.

## Acceptance
- The human edits the parameter matrix in the UI; the next run's work-list actions, dispute
  admissibility, arbiter escalation, and cohort alerts all reflect it.
- The governing matrix (version + values) is recorded in the run's audit trail.
- Named matrices persist and recall; the built-in default reproduces today's behaviour exactly.
- Invalid inputs are rejected with a helpful message; **offline/tests unchanged** when no matrix is
  supplied (defaults to the current constants).

## Out of scope
- The **per-dispute** approve/reject gate — already exists (`ApprovalGate`); this is the *policy* the
  gate and agents operate under, set up front.
- **Auto-tuning** the matrix from outcomes (a later, data-driven step — needs WA-093's monitoring).
- Changing model internals — policy only, not the PD model (WA-094).
