# WA-070 · Pre-submission review: features, architecture, requirements

**Candidates:** Stefanie (EM review) or Claude. Read-only; produces a gap list, not fixes.

## Goal

One honest pass over the whole product against (a) the HACKATHON.md rubric
mapping and (b) the submission requirements, answering: does every claimed
feature actually work, and does the architecture story hold together for a
read-in judge?

## What to review

1. **Feature claims vs reality.** Walk every ✅ row in README's Status table
   and HACKATHON.md's rubric mapping. For each: does it work end-to-end today?
   Cross-check against Bimo's WA-064 findings + WA-069 report first — several
   verdicts already exist (no P0s; WA-047 doc drift; live qwen path
   unverified).
2. **Architecture coherence.** Does the documented architecture (two tiers,
   6-agent society, OSS lakehouse, RDS auth, FC deployment) match what's
   actually in the code and IaC? Flag any doc-vs-reality divergence — judges
   read the docs then poke the system.
3. **Requirements coverage.** The submission checklist (see WA-076): public
   repo + license, Alibaba Cloud proof link, architecture diagram, 3-min
   video, text description, track ID. For each: already satisfied / partially /
   missing — and which ticket closes it.
4. **Known red flags.** Live `brain=qwen` debate times out at 120s on prod
   (WA-069 finding) — the demo-critical open risk. Custom domain (WA-067)
   in flight. Assess impact on the demo story.

## Acceptance

`backlog/WA-070-review.md`: per-area verdict (solid / gap / broken), each gap
with a suggested owner + effort estimate, and a final "submission-ready /
ready-with-fixes / not-ready" call with the must-fix list. No code changes.

## Out of scope

- Fixing anything found (fixes get their own tickets).
- Re-running WA-064/069 verification — cite them, don't redo.
