# WA-029 · DuckDB + dlt lakehouse data engine

## Goal

Replace the bulk-download data layer with a proper data lakehouse pattern:
dlt reads Parquet from OSS (S3-compatible) → loads into DuckDB → agents query
via SQL instead of in-memory Python. RDS PostgreSQL federates for operational
data (auth, disputes, audit). This is the architecture upgrade that signals
sophistication to judges (Criterion 2: "high-quality architecture, thoughtful
adoption").

## Architecture change

```
BEFORE:
  OSS → bulk download → in-memory Arrow → Python features → score

AFTER:
  OSS (loans.parquet)
    ↓ dlt filesystem source (S3-compatible, incremental + merge)
  dlt pipeline (schema contracts + dedup)
    ↓ load to DuckDB (analytics) + PostgreSQL (operational)
  DuckDB query engine (agents run SQL, not Python)
    + postgres_scan federation (join OSS data with RDS tables)
```

## Implementation (3 layers)

### Layer 1: dlt ingestion pipeline (replaces oss.py bulk download)
- `waspada/lakehouse/pipeline.py` (NEW) — dlt pipeline that:
  - Reads from OSS via `filesystem(bucket_url="s3://waspada-loans/", endpoint_url=...)`
  - Uses `read_parquet()` from dlt.sources.filesystem
  - Loads into DuckDB destination (`waspada.duckdb` file, in-process)
  - Schema contract: freeze the RawLoans shape, reject drift
  - Incremental: `dlt.sources.incremental` on a cursor column (for re-runs)
  - `load_info` returned with row count, load time, schema version
- Keep `waspada/data/oss.py` as fallback when dlt/DuckDB unavailable
- The IngestAgent's `fetch` tool can use either path (dlt preferred, OSS fallback)

### Layer 2: DuckDB query layer (replaces Python analytics)
- `waspada/lakehouse/duckdb.py` (NEW) — DuckDB connection manager:
  - In-process DuckDB instance (persisted to `data/waspada.duckdb`)
  - `query(sql) -> pa.Table` — run SQL, return Arrow
  - `table_exists(name) -> bool`
  - `table_stats(name) -> dict` (row count, column types)
- The MCP server's `portfolio_stats` and `lookup_account` tools become
  DuckDB SQL queries instead of Python aggregation over Arrow tables
- The AnalyticsAgent's feature build can optionally use DuckDB SQL
  (`SELECT loan_id, total_paid/amount AS payment_ratio, ...`)

### Layer 3: PostgreSQL federation (cross-source queries)
- DuckDB's `postgres_scanner` extension reads from RDS PostgreSQL
- Enables: `SELECT * FROM duckdb_loans l JOIN postgres_disputes d ON l.loan_id = d.loan_id`
- Dispute memory (WA-026) migrates from JSON file to RDS table
- The agents can query both sources in one SQL statement

## Dependencies

- `dlt[duckdb]>=1.0` — the pipeline library + DuckDB destination
- `duckdb>=1.0` — the query engine (Python binding)
- Both are pip-installable, run in-process (no daemon), work inside Docker/FC

## Acceptance

- dlt pipeline loads loans.parquet from OSS into DuckDB (or local file fallback)
- DuckDB query layer returns correct results for portfolio_stats + lookup_account
- MCP tools work via DuckDB queries (same output, different engine)
- Schema contract rejects a drifted Parquet (wrong column types)
- Local fallback works without OSS creds (load from local file)
- All existing tests green; new tests cover the DuckDB query layer
- The old OSS path (oss.py) stays as fallback, not removed

## Out of scope

- AI Data Engineer / Data Analyst agents (WA-030 — separate ticket)
- OSS S3 endpoint verification (needs Alibaba creds — stub for now)
- Remote DuckDB / MotherDuck (in-process is sufficient)

## Notes for the worker

- Install: `pip install dlt[duckdb] duckdb`
- DuckDB reads Parquet natively — you can test WITHOUT OSS by pointing
  at a local file: `read_parquet('data/loans.parquet')`
- The dlt filesystem source supports S3-compatible endpoints via `endpoint_url`
- For Alibaba OSS: `bucket_url="s3://waspada-prod-loans/"`,
  `endpoint_url="https://oss-ap-southeast-1.aliyuncs.com"`
- DuckDB in-process means no server to manage — it's just a Python library
- Keep the connection as a module-level singleton (not per-request)
