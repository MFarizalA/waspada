# WA-057 · Three-bucket medallion data architecture (Raw → Staging → Mart)

## Goal

Replace the single-bucket OSS design with three physically isolated buckets
following the medallion pattern: **Raw** (immutable source), **Staging**
(lane-specific curated views), **Mart** (serving layer for agents + dashboard).

Three buckets instead of one gives us: separate access policies per zone,
different lifecycle rules, a write boundary that prevents agents from corrupting
source data, and a cleaner architecture story for judges.

## Bucket layout

```
waspada-raw/             ← Bronze — immutable source
└── loans.parquet

waspada-staging/         ← Silver — lane-specific curated views
├── collections/
│   ├── feature_frame.parquet
│   └── quality_report.json
└── origination/
    └── feature_frame.parquet

waspada-mart/            ← Gold — serving layer
├── collections/
│   ├── scored_accounts.parquet
│   ├── dashboard_payload.json
│   └── debate_transcript.json
└── origination/
    ├── scored_accounts.parquet
    └── dashboard_payload.json
```

## IaC changes (`deploy/iac/main.tf` + `variables.tf`)

Replace the single `alicloud_oss_bucket.loans` + ACL with three buckets:

| Resource | Name | ACL | Purpose |
|---|---|---|---|
| `alicloud_oss_bucket.raw` | `waspada-prod-raw` | private | Immutable source — owner upload only |
| `alicloud_oss_bucket.staging` | `waspada-prod-staging` | private | Curated lane views — pipeline writes |
| `alicloud_oss_bucket.mart` | `waspada-prod-mart` | private | Serving layer — orchestrator writes |

Each gets its own ACL resource (split, per provider convention).

RAM policy: FC execution role gains read on Raw + read/write on Staging + Mart.
No write access to Raw for any compute role — only the owner uploads.

## Code changes

| Component | Change |
|---|---|
| `config.py` | `OSS_BUCKET` → `OSS_RAW_BUCKET`, `OSS_STAGING_BUCKET`, `OSS_MART_BUCKET` |
| `waspada/data/oss.py` | `OSSClient` reads from Raw bucket only |
| `waspada/agents/data_analyst.py` | Optional: write FeatureFrame to Staging after build |
| `waspada/agents/orchestrator.py` | Optional: write scored output + payload to Mart |
| `.env.example` | Document all three bucket vars |
| `deploy/iac/main.tf` | Three buckets + ACLs + RAM policy update |

## Acceptance

- Three OSS buckets created by `tofu apply`.
- Raw bucket: read-only for FC execution role.
- Staging + Mart buckets: read/write for FC execution role.
- `OSSClient.fetch_loans()` reads from Raw.
- `.env.example` documents all three vars.
- Existing tests green (env var names change but offline path unaffected).

## Out of scope

- DuckDB-native OSS reads (WA-055 — separate ticket).
- Lane-aware column projection (WA-034/035 — Origination lane).
- Data Analyst live MCP service (WA-056 — separate ticket).
- Lifecycle policies beyond default (versioning, TTL — post-hackathon).

## Notes for the worker

- IaC changes are straightforward — same pattern as the existing bucket, ×3.
- The code changes (`config.py`, `oss.py`) are small — just env var renaming.
- The optional Staging/Mart writes (feature_frame persist, scored output persist)
  can be additive and gated behind a feature flag if time is tight.
- For the hackathon demo, only the Raw bucket needs data. Staging and Mart are
  created empty — they're architecture, not a deploy blocker.
