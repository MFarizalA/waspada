# WA-046 · BUG: cross-run dispute memory never persists (WA-026 is inert in production)

## The bug

WA-026 shipped the dispute-memory machinery, but **no entrypoint wires a
persistent backend** — so cross-run memory does not survive a process restart
anywhere in the running system. The headline capability it was built for does
not actually work.

Evidence (verified 2026-07-14):

- `api/main.py:122-128` — `Orchestrator(llm, as_of=…, top_n=20,
  on_round_complete=…, on_dispute_resolved=…)`. **No `memory=` / `memory_backend=`
  argument.**
- `waspada/agents/__main__.py:109` — `Orchestrator(get_llm(), as_of=as_of,
  top_n=args.top_n)`. **Same — no memory argument.**
- `waspada/agents/dispute_memory.py:190-191` — `DisputeMemory.__init__` therefore
  falls back: `self.backend = backend if backend is not None else InMemoryMemory()`.
- `waspada/agents/orchestrator.py:110-111` — the comment **claims** *"the API
  wires a LocalFileMemory"*. It does not. This stale comment is why the gap went
  unnoticed.
- `LocalFileMemory` is referenced only in `dispute_memory.py`, `agents/__init__.py`
  (export), `orchestrator.py` (type/comment), and `tests/` — **never in a real
  entrypoint.**

## Why it matters (not cosmetic)

The whole point of WA-026 is the **second efficiency axis**: *"run the same book
twice → run 2 spends measurably fewer LLM calls + fewer escalations"*
(HACKATHON.md:476-477), and the staged run-1/run-2 demo. With in-process memory
only, **run 2 starts empty every time** — precedent is never applied, no dispute
is ever short-circuited, and the efficiency claim cannot be reproduced in the
deployed app. `DisputeMemory.persist()` runs and writes nothing durable.

## Decision (owner, 2026-07-14): dispute memory lives in **OSS**, not RDS

HACKATHON.md contradicts itself — `:467` says disputes persist to **OSS**
(`disputes/` index keyed by `loan_id`); `:320-321` lists dispute memory under
**RDS PostgreSQL**. **Resolved: OSS.** The lakehouse is OSS + DuckDB only
(see WA-047); RDS stays auth-only. Fix the HACKATHON.md contradiction as part
of this ticket.

## Files / interfaces

- **Fix the wiring (the actual bug — do this even if the OSS backend slips):**
  - `api/main.py` `_build_demo_orchestrator` and `waspada/agents/__main__.py` —
    pass a real backend: `Orchestrator(..., memory_backend=get_memory_backend())`.
  - Correct the stale comment at `orchestrator.py:110-111`.
- **New `OSSMemory(MemoryBackend)`** in `waspada/agents/dispute_memory.py` —
  drops in behind the existing 2-method ABC (`load_memory()` / `save_memory(dict)`,
  `dispute_memory.py:77-94`); the module already anticipates it (`:39-41`, `:81-83`:
  *"An OSS backend (deferred) implements the same two methods against `oss2` and
  needs no orchestrator changes"*). **No orchestrator changes needed.**
- **Backend selection helper** — OSS when creds are configured, else
  `LocalFileMemory` (mirrors the `get_audit_sink()` fail-safe pattern in
  `waspada/audit/sls.py:168-170`). Never `InMemoryMemory` in a real entrypoint.

## Storage shape — per-`loan_id` objects, NOT one blob

`LocalFileMemory` overwrites a single whole `data/dispute_memory.json` on every
save (`:114-159`). Ported naively to OSS that becomes a **lost-update hazard**:
Function Compute runs with `instance_concurrency = 10`
(`deploy/iac/main.tf`), so concurrent invocations would read-modify-write the
same object and silently drop each other's disputes.

Use one object per account: `memory/disputes/loan_id={id}.json` (see WA-047's
bucket layout). Volume is tiny — bounded by the audit top-K, *"typically ≤ 16"*
per run (`dispute_memory.py:82`).

## Acceptance

- A real entrypoint (CLI **and** API) persists resolved disputes durably; a
  second run on the same book **loads the prior run's memory** — assert
  `short_circuited` / `precedent_hits` counters are non-zero on run 2.
- Regression: run-1 → restart process → run-2 short-circuits at least one
  account with a prior **human** ruling (the `_SHORT_CIRCUIT_BY = "human"` rule,
  `dispute_memory.py:188`).
- Concurrent writes don't lose disputes (per-loan objects; no whole-index
  overwrite).
- Fail-safe: OSS unavailable → falls back to local file, run never breaks
  (same discipline as the audit sink).
- Offline tests green on the existing `InMemoryMemory`/`LocalFileMemory` paths.

## Depends on
