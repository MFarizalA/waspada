# WA-093 · PD model monitoring & drift observability

**Candidates:** Claude or data/model lane. Prompted by the new **Lending Club** data source: with a
real, shifting book we need to *see* drift, not assume it away. Strong governance story for the
Agent-Society submission.

## Why (grounded in how the model actually works)
The PD model is a `LogisticRegression` **fit fresh every run** on the incoming `FeatureFrame`
(`waspada/model/risk.py::train`, vintage split, hold-out AUC). So the classic failure mode — a
*stale* model silently degrading — **structurally can't happen**; every run retrains. Two guards are
already correct by construction: the label (`is_default()` / `DEFAULT_STATUSES`, shared by synthetic
+ LC) and the leakage-safe `FEATURE_COLUMNS` subset.

**But fit-per-run has its own blind spots, and a new source makes them bite:**
1. **Covariate shift is invisible** because `predict()` bands are **per-batch relative** (quintiles
   of the current run) — if the LC book is systematically riskier/safer, the dashboard still shows
   ~20% "Very High." The band mix never moves even when the portfolio does.
2. **No cross-run drift metric.** Each run computes hold-out AUC + can compute the observed default
   rate, but nothing is persisted or compared across runs, so there's no drift timeline.
3. **Batch instability.** A small/skewed batch (or a single-vintage cohort → the shuffle-split
   fallback) yields a jumpy model with no visible warning.

## Design
- **Per-run model-monitoring record** (emit + persist): hold-out `AUC` (already computed), **observed
  default rate** (`mean(label_default)`), train/test row counts + split method, and the **band
  distribution** (share per RISK_LEVEL). One JSON record per run.
- **Population Stability Index (PSI)** per feature vs a **stored reference cohort** (a baseline
  snapshot — e.g. the first LC load, or a pinned reference parquet). Flag `PSI > 0.2` (moderate) /
  `> 0.25` (significant) per feature; surface the top drifted features.
- **Persist the records** so they're comparable across runs — OSS (a `monitoring/dt=<batch>/…` Gold
  sibling) and/or the existing SLS audit sink — giving a **drift timeline**.
- **Serve absolute bands by default.** WA-051 already added absolute `band_edges`; compute them once
  from the reference cohort and use them for serving so the **band mix moves when the book moves** —
  the single change that makes covariate shift *visible* on the dashboard.
- **Surface it.** A small "model card / monitoring" strip (AUC, default rate, drift flags) in the
  dashboard audit area — and cite it in the debate as data-trust evidence.

## Acceptance
- Every run emits a monitoring record (AUC, observed default rate, split sizes, band distribution)
  and per-feature PSI vs the reference cohort; significant drift is flagged.
- Records persist and are queryable across runs (a drift timeline exists).
- Serving uses absolute band edges so the band distribution reflects real portfolio shift.
- Fully **offline-safe/guarded**: no reference/OSS configured → monitoring degrades to log-only, the
  pipeline is byte-for-byte unchanged (same pattern as the SLS sink / medallion writes).

## Out of scope
- Model **versioning/persistence** mechanics — that's **WA-082**.
- Improving the model itself (calibration, train/serve decoupling, upgrade) — that's **WA-094**.
- Automated retraining triggers off the drift signal (a later step once the signal exists).

## Notes
- Pairs naturally with **WA-094** (a calibrated, decoupled model makes AUC + default-rate tracking
  meaningful) and **WA-082** (a versioned model lets a monitoring record name *which* model drifted).
