# WA-078 · Land a real Lending Club dataset in OSS as the RawLoans source

**Owner directive (2026-07-19):** no more fake data. The raw bucket is empty;
this ticket puts a real, RawLoans-conformant parquet file in it.

## Goal

Source the public Lending Club accepted-loans dataset, map it to the frozen
`RawLoans` schema, write parquet, and upload to
`oss://waspada-prod-raw/<the configured key>` so the WA-077 cutover has real
data to read.

## Constraints (from memory + schema — VERIFY against code, don't trust this list)

- **Cross-sectional snapshot, NOT a monthly panel.** One row per loan,
  `as_of` snapshot semantics.
- **Collections label = eventual default.** Map `loan_status` → the label
  the way `features/collections.py` / the status-mapping logic expects
  (Charged Off / Default → 1; Fully Paid → 0; drop Current/Late-ambiguous
  per the existing mapping code — READ the code, don't invent).
- **No roll rates.** Schema is in `waspada/data/schema.py` (or wherever
  RawLoans + `_RAW_LOANS_COLUMNS` live — `oss.py` selects exactly those
  columns in declaration order; the parquet must carry them all).
- Keep it lean: subsample to ~10-50k rows if the full LC file is huge —
  enough for real distributions and visible debates, small enough for FC's
  bulk-download read (WA-047 note: reads are whole-object, no pushdown).

## Method

1. Read `waspada/data/schema.py`, `waspada/data/oss.py` (`_RAW_LOANS_COLUMNS`),
   and `features/collections.py` for the exact columns + dtypes + the
   delinquency-status mapping. The `validate_table(table, RawLoans)` call in
   `fetch_loans` is the gate — your file must pass it.
2. Download Lending Club accepted loans (Kaggle public dataset; if Kaggle
   auth is a pain, use any mirror of `accepted_2007_to_2018Q4.csv` — note
   the source in the commit).
3. Transform script → `scripts/build-real-rawloans.py` (committed, reusable):
   LC columns → RawLoans schema (loan_id, amount, outstanding_ratio or
   outstanding_principal per schema, delinquency status, label, vintage/as_of
   fields per schema). Handle missing values honestly (drop or impute per
   what the schema validation allows — no fabricated values).
4. Validate locally: `validate_table` passes; `oss.fetch_loans` against the
   uploaded file returns a clean table; the full pipeline runs on it locally
   (orchestrator completes, agents ok).
5. Upload: use the deploy creds pattern (the same AccessKey the app uses —
   it's in `.env` / secrets.tfvars; if your profile's aliyun CLI key 403s on
   the bucket, use oss2 from a Python script with the .env creds — do NOT
   print secrets).
6. Verify in OSS: object exists, size sane, `fetch_loans` from the real
   bucket returns rows.

## Acceptance

- `scripts/build-real-rawloans.py` committed and reproducible.
- Real parquet in `oss://waspada-prod-raw/` at the configured key; the app's
  `fetch_loans()` returns a validated table from it.
- Local full-pipeline run on the real file completes with all agents ok.
- A note in the commit/report: row count, default rate, date range of the
  data (judges may ask).

## Out of scope

- The API cutover (WA-077 — parallel ticket, lands after this one).
- Origination lane data.
- Monthly-panel anything (we are cross-sectional by design).
