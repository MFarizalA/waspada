# WA-048 · Close the loop: the adjudicated band drives the work-list

## The bug (architectural, not cosmetic)

**The debate changes nothing.** The society produces a transcript beside the
decision, not a decision.

Verified 2026-07-15:

- `data_handles["scored_accounts"]` is assigned in **exactly one place** —
  `risk_model.py:88-89`. Nothing after the debate ever touches it.
- `ranking.rank()` derives `recommended_action` **purely from `score_band`**
  (`ranking.py:105`, `ACTION_BY_BAND`).
- `insight.py:104` reads `dispute.resolution` **only** to test OPEN-ness
  (`open_disputes = [d for d in disputes if not d.resolution]`). It is never
  applied to alter an action or a band.
- `Dispute` (`protocol.py:183-190`) has **no field capable of carrying a revised
  band**. `model_band` is the Actuary's *original* band; `auditor_view` is the
  Skeptic's *opening* read. Neither is written by the debate.

Consequence: when a dispute resolves as **`overridden` — meaning the Actuary
itself CONCEDED** — the account still ships with its **original band and
original action**. The analyst is told "call this account," and shown, in a
separate panel, a transcript in which the model admitted it was wrong.

Until a resolution feeds back into the band/action, the Agent Society is
**narratively load-bearing but operationally inert**.

## Decision (owner, 2026-07-15)

### 1. `p_default` and `score_band` are immutable facts

The debate **never** rewrites them. The society writes a **new `final_band` +
`override_reason`**; `recommended_action` derives from `final_band`.

This is the credit-ops pattern: the model score stands and stays auditable, and
the override is a **documented, reason-coded policy layer** on top. It also lets
us *measure* society-vs-model divergence — impossible if the debate overwrites
the number.

```
scored_accounts, after the debate

loan_id  p_default  score_band   final_band  action   override_reason
LN0042      0.91    Very High    Medium      watch    arbiter: 4 on-time payments
                                                      post-delinquency
LN0107      0.88    Very High    Very High   call     (upheld)
```

### 2. Asymmetric governance by direction

The error economics of collections are asymmetric, so the governance is too:

| Direction | Failure mode | Cost | Rule |
|---|---|---|---|
| **Escalation** (society raises risk) | a wasted collector call | cheap | **auto-apply** |
| **De-escalation** (society cancels a call) | a real default walks away | expensive | **human sign-off** |

De-escalations route to the `ApprovalGate` and apply **only** on
`escalated_approved`. The protocol already carries `escalated_approved` /
`escalated_rejected` (`orchestrator.py:549-551`) — this finally gives them a job.

## Files / interfaces

1. **`waspada/agents/protocol.py`** — add `revised_band: str = ""` and
   `applied: bool = False` to `Dispute`. Defaults keep every existing
   construction site valid. Extend `to_dict()` (currently 8 keys, `:203-217`).
2. **`waspada/agents/arbiter.py`** — the ruling JSON gains `revised_band`,
   validated against `schema.RISK_LEVELS`. On `"override"` it must *name* the
   band. Fallback when absent: project `auditor_view` (`Low|Medium|High`) onto
   `RISK_LEVELS`.
3. **`waspada/agents/risk_model.py`** — on the concede path
   (`orchestrator.py:406-408`), conceding means the Skeptic's view stands →
   `revised_band` from `auditor_view`.
4. **`waspada/agents/orchestrator.py`** — new `_apply_adjudications(ctx)`, called
   right after `_resolve_disputes` (existing call site `:260-261`). Appends
   `final_band` + `override_reason` columns and **re-assigns**
   `ctx.data_handles["scored_accounts"]`. Reuse the append-column pattern already
   proven at `risk.py:342-353`. The direction rule lives here.
   Derive the ordinal from `schema.RISK_LEVELS` rather than re-hardcoding it —
   `risk_auditor.py:49-51` duplicates that order today and should import it too.
5. **`waspada/insight/ranking.py`** — `rank()` reads `final_band` via the existing
   `_safe_get` additive probe (`:95`/`:124` — the same mechanism
   `outstanding_principal`/`expected_loss` already uses). Action becomes
   `ACTION_BY_BAND[final_band or score_band]`. Emit `score_band`, `final_band`,
   and `override_reason` on the row.
6. **`waspada/schema.py`** — document `final_band` / `override_reason` as additive
   optional, following the `agent_dialogue` precedent (`:153-166`).
   **No contract break:** `validate_table` explicitly allows supersets
   (`:219-220`).
7. **Dashboard** — work-list shows the model band struck through → final band,
   with the override reason on hover. Extend `dashboard/src/lib/riskLevel.ts`;
   add EN/中文 i18n keys.

## Acceptance

- A run in which the Actuary concedes on a `Very High` account the Skeptic views
  as `Low` produces a work-list row whose **`recommended_action` has actually
  changed** — and changed **only after** a human approves, because that is a
  de-escalation. **Assert on `payload["work_list"]`, never on `agent_dialogue`** —
  the whole point is that the transcript is no longer the only artifact.
- An escalation applies with **no** gate call.
- A de-escalation is withheld on `escalated_rejected`.
- `validate_table(scored, ScoredAccounts)` still passes with the two new columns.
- Offline suite green (MockLLM, no network).
