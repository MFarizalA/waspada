# WA-082 · ML governance — persist + version the PD model binary in OSS

**Owner:** Claude. **Depends on:** an OSS write helper (shared prerequisite — see Notes).
**Rationale + the "why NOT PAI-EAS" research:** [`backlog/pd-model-hosting.md`](pd-model-hosting.md).

## Goal
Score with a **frozen, versioned, auditable** PD model instead of a subtly-different in-process
re-fit each run. Reproducible, defensible decisions — *"this run scored with model `pd-lr-<id>`,
trained as-of T, AUC=Y"* — a governance/audit asset that fits the society's audit ethos. Stays
**in-process ($0, no PAI-EAS)** and **keeps `explain()` intact** (the model internals the Risk
Auditor cites in the debate).

> Not a speed play — a `LogisticRegression.fit` is fast; the retrain saving is marginal. The value
> is **reproducibility + auditability + versioning**, not latency.

## What to do
1. **OSS write path (prerequisite).** `oss.py` is **read-only today** (no `put_object`; the
   medallion RAM policy grants PutObject on staging/mart but no code uses it). Add an upload helper.
   **Prefer generalizing it into the shared write path** (also unblocks dlt Option-B, medallion
   feature writes, and the `dispute_memory.py` `TODO(WA-047) add OSSMemory` when the policy allows
   PutObject) — build once, unblock four. If the shared path isn't ready, ship a minimal
   model-scoped write as an interim and note it for consolidation.
2. **Versioned artifact.** Extend the existing pickled model dict (already `explain()`-compatible:
   `coef_`, `band_edges`, feature list) with a lineage header — `model_id` (`pd-lr-<timestamp|sha>`),
   `trained_as_of`, `n_train_rows`, AUC/metrics, feature list, schema/contract version. Store at
   `oss://<staging|mart>/models/pd/<model_id>.pkl` + a `latest.json` manifest pointer.
3. **Publish step (train offline).** A CI/manual entrypoint fits the model and uploads the versioned
   binary + `latest` pointer. **Not per-run.**
4. **Serve wiring (`risk_model` agent).** On run, `load_model` from OSS (`latest`, or a pinned
   `model_id`) when configured/present; else **fall back to train-per-run** (offline / tests / demo
   unchanged). Scoring + `explain()` unchanged.
5. **Audit / governance stamping.** Put `model_id` + `trained_as_of` + metrics into the run's step
   log / SLS audit + the dashboard payload, so **every decision cites the exact model version.**

## Acceptance
- A versioned model binary is published to OSS and loaded at serve-time; the run scores with it.
- `explain()` / driver evidence works unchanged on the loaded model (the debate is unaffected).
- Every run's audit trail + dashboard payload cite `model_id` + `trained_as_of`.
- **Train-per-run fallback preserved:** with no OSS/model configured, offline runs and the full test
  suite stay green (byte-for-byte).
- Tests: publish→load round-trip; the fallback path; audit-stamp presence.

## Out of scope
- **PAI-EAS / external model serving** — researched and rejected (per-minute dedicated instance, no
  scale-to-zero for custom models, breaks `explain()`; see `pd-model-hosting.md`).
- Retraining-cadence *automation* — a manual/CI trigger is fine; document the refresh policy.
- Generalizing the OSS write path beyond what this needs — coordinate the shape with the data lane;
  this ticket only requires the model-artifact write.
