# WA-084 · Data Engineer + Data Analyst use NATIVE function calling (conformance A4)

**SHIPPED** — `feature/a4-native-function-calling` (`dfa4edf`). Closes gap A4 from
`backlog/architecture-conformance-review.md`.

## What it was
The docs claimed all three loop agents "use the same Qwen native function-calling
loop," but only `risk_auditor` did. `data_engineer` and `data_analyst` drove their
loops with `llm.complete()` + regex-parsing a `{"tool","arg"}` blob from free text —
a prompt-embedded convention, not the native `chat()`/`tools`/`tool_calls` surface.
This over-claimed on a judging-rubric axis (sophisticated native function calling).

## What shipped
- `llm.py`: explicit `supports_native_tools` capability flag (base `False`,
  `QwenLLM=True`, `MockLLM` opts in via `native_tools=`). Dispatch reads the flag so a
  scripted MockLLM is identified as non-native **without consuming a scripted reply**.
- `data_engineer.py` / `data_analyst.py`: `_reasoning_loop` dispatches native-vs-legacy;
  `_native_reasoning_loop` runs `chat(tools=…)` with tool-role feedback; returns `None`
  on the first turn if no native `tool_calls` → transparent fallback to the legacy loop.
  Native tool schemas (`_DE_TOOLS` / `_ANALYST_TOOLS`) declared.
- Offline/tests byte-for-byte unchanged (scripted mocks stay on the legacy path). +3
  native tests. 446 pass (443 baseline + 3); the 2 `test_stream` failures are the
  pre-existing WA-077 SSE tests (Bimo's), unrelated.
