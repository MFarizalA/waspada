## Architecture decision (owner, 2026-07-14)

**The lakehouse is OSS (storage) + DuckDB (engine). Nothing else.**

- **RDS PostgreSQL is NOT part of the lakehouse** — it stays the operational
  auth store (WA-028) and nothing more.
- **DuckDB↔RDS federation via `postgres_scanner` is DESCOPED**, not deferred.
  Remove the 🟡 claim from HACKATHON.md rather than carrying it as debt.
- **Dispute memory lives in OSS**, resolving HACKATHON.md's self-contradiction
  (`:467` OSS vs `:320-321` RDS). See WA-046.

This ticket makes the lakehouse claim honest: today it is a **data lake read**,
not a lakehouse — no table format, no versioning, no snapshot isolation.

## What's actually true today (verified 2026-07-14)

- **One flat object.** `OSS_KEY` is a *fully-qualified object key*
  (`config.py:74`), so the lane cannot vary the path — `oss.py:100-119` validates
  `lane` then reads the **same object for both lanes** (`WA-038.md:26-28` confirms).
- **Bulk download, zero pushdown.** `oss.py:113-117` — `get_object().read()` →
  `pq.read_table()` → client-side `limit` slice. Agents then build an in-memory
  DuckDB over the Arrow table (`data_engineer.py:136`, `_arrow_lakehouse`
  `:393-399`). **No `httpfs`, no `read_parquet('s3://…')` anywhere.**
- **The dlt path is dead code with TWO defects** (`lakehouse.py:126-152`):
  (1) `dlt.readers.filesystem` **does not exist** (`:145`); (2) even if fixed,
  `pipeline.run()` writes to its *own* DuckDB file while `load_to_duckdb()`
  returns a **different, empty** `con` (`:110`, `:152`) — the "reopen read-only"
  comment is an unwritten TODO. **Nothing calls `load_to_duckdb` at all.**
- **The bucket is read-only** — RAM policy grants `GetObject`/`GetObjectMeta`/
  `ListObjects`/`GetBucketInfo` only (`deploy/iac/main.tf:86-107`). **No writes
  exist anywhere in the codebase** (zero `put_object`).
- **No ingest/upload script exists** (`scripts/` has only `README.md` + `wt.sh`;
  WA-027 left the upload manual).
- **Scale: 33 MB / 1M rows.** Partitioning buys **nothing for performance** — it
  is for **versioning, lane separation, and lineage**. Do not over-engineer.

## Bucket layout

```
waspada-{env}-loans/
├── raw/lane=collections/as_of=YYYY-MM-DD/loans.parquet
├── raw/lane=origination/as_of=YYYY-MM-DD/applications.parquet   (WA-034)
├── serving/lane={lane}/as_of=YYYY-MM-DD/
│   ├── scored_accounts.parquet
│   └── payload.json
│   └── (serving/lane={lane}/latest.json → pointer for the dashboard)
├── memory/disputes/loan_id={id}.json        (WA-046 — per-loan, not one blob)
├── models/lane={lane}/trained_as_of=YYYY-MM-DD/model.pkl + metrics.json
├── audit/dt=YYYY-MM-DD/{run_id}.jsonl       (cold tier behind SLS)
└── bench/{run_id}.json
```

**`as_of=` is the load-bearing partition.** `RawLoans` has **no snapshot-date
column** — `as_of_date` exists only downstream on `FeatureFrame` as a
*caller-supplied parameter* (`schema.py:92`; `api/main.py:124` passes it in). So
the snapshot date is metadata **only the object path can carry**. The book is
re-scored daily with the same `loan_id`s but *different* `current_status`
(HACKATHON.md:468), so an as-of prefix is the only correct time axis.

Accumulating dated snapshots is also what would eventually **lift the honesty
rail** — band migration and cure rates (today explicitly *"NOT computable from
one snapshot — do not fake them"*, HACKATHON.md:355-358) become genuinely
computable. Do not claim them until multiple snapshots exist.

**Do NOT partition by `current_status`** — it is mutable across snapshots, so
partitions would rewrite themselves. It stays a predicate column.

**`lane=` as a hive partition, not a second env var.** This resolves the fork
WA-038 leaves open (`WA-038.md:39-40`: *"add `OSS_ORIGINATION_KEY` **or a
lane-suffixed convention**"*) — the partition scales to N lanes without an env
var each, and `.env.example:11`'s own example (`collections/loans.parquet`)
already implies it.

**No Iceberg/Delta.** At 33 MB with a daily full refresh, hive-partitioned
Parquet gives versioning and lane separation without a table format's
operational weight. Revisit only if true concurrent ACID writes become real.

## Files / interfaces

1. **Env contract: `OSS_KEY` (object key) → `OSS_PREFIX` (prefix) + a resolver.**
   This is the migration that unblocks everything. Add
   `waspada/data/paths.py` (or extend `config.py`) with e.g.
   `raw_key(lane, as_of) -> "raw/lane={lane}/as_of={as_of}/loans.parquet"`.
   **Keep `OSS_KEY` working as an explicit override** so `oss.py:113` doesn't
   break on day one (back-compat: if `OSS_KEY` is set, use it verbatim).
2. **`waspada/data/oss.py`** — lane-aware key resolution + per-lane column
   projection (`RawLoans` vs `RawApplications`, per WA-038).
3. **Read path — pick ONE and make it real:**
   - **(a) Recommended: DuckDB `httpfs`** — `INSTALL/LOAD httpfs`, `SET s3_endpoint`,
     then `read_parquet('s3://…')`. Direct, no dlt dependency, and gives genuine
     pushdown/partition pruning. `_oss_s3_endpoint()` (`lakehouse.py:67-83`)
     already maps `OSS_*` → S3-compatible settings.
   - **(b) Or fix dlt** — must fix *both* defects (real
     `dlt.sources.filesystem` API **and** land the data in the returned `con`),
     plus add the incremental cursor + `merge` on `loan_id` that HACKATHON.md
     already claims.
   Either way, **delete the dead `source_url` (`lakehouse.py:138`) and the no-op
   `import dlt as _dlt` (`:151`)**, and make `load_to_duckdb` actually get called
   — or delete it.
4. **IaC — add `PutObject`** to the FC RAM policy (`main.tf:86-107`). Deliberate
   security change: the bucket is read-only today and every write-side artifact
   (disputes, models, serving, audit) needs it.
5. **`scripts/upload_loans.py`** — the missing ingest writer (there is none).
6. **Bucket storage policy** (currently: private ACL + read-only RAM, and
   *nothing else*): add **versioning** (an overwrite of `loans.parquet` is
   unrecoverable today), a **lifecycle rule** (audit/ → IA/Archive; storage class
   defaults to Standard), and consider **access logging** — we audit every agent
   turn to SLS but have **zero record of who reads the loan book**.
7. **HACKATHON.md** — remove the `postgres_scanner` federation 🟡 (descoped);
   correct the ✅ on *"dlt filesystem source … incremental loading + merge dedup"*
   and *"DuckDB reads Parquet directly via `httpfs`"* — **neither is implemented**;
   fix the dispute-memory OSS-vs-RDS contradiction → OSS.

## Acceptance

- A lane+as_of resolves to a real object; both lanes read their own data.
- DuckDB queries the OSS Parquet **without a full client-side download** (or, if
  dlt: the returned `Lakehouse` actually contains the loaded table — the current
  bug).
- `PutObject` granted; an upload script populates `raw/lane=collections/as_of=…/`.
- Back-compat: an explicitly-set `OSS_KEY` still works; offline paths (in-memory
  Arrow / local Parquet) unchanged; full suite green.
- HACKATHON.md's lakehouse section matches reality.

## Out of scope

- RDS federation / `postgres_scanner` — **descoped, not deferred.**
- Iceberg/Delta/Hudi table formats.
- Multi-snapshot analytics (band migration, cure rates) — unlocked by `as_of=`
  but a separate ticket, and **not claimable until multiple snapshots exist**.
