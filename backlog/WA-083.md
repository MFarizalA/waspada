# WA-083 · Utilize dlt in WASPADA — a real load/write pipeline

**Candidates:** Claude (did the WA-047 dlt research + PoC) or the data lane (Bimo = Data Engineer;
Stefanie = load/write layer).

**Reverses** WA-047's 2026-07-14 "the lakehouse is OSS + DuckDB, **nothing else** / no dlt pipeline"
stance — an **owner-directed decision (2026-07-19)** to make dlt *real*, backed by the WA-081
"dlt stays" call and the **PoC-proven feasibility** in [`backlog/WA-047-dlt-research.md`](WA-047-dlt-research.md).
(The earlier dlt scaffold was correctly removed — it used a nonexistent `dlt.readers.filesystem`;
the real API is `dlt.sources.filesystem`, verified working.)

## Goal
Turn "we use dlt" from a **declared-but-unused dependency** (a WA-070 doc-vs-reality exposure) into
a **real, demoable pipeline**: dlt loads + validates the loan book (schema contract, merge dedup,
lineage) and writes the Data Analyst's derived features to the OSS medallion. Closes two
doc-vs-reality gaps and gives the Data Engineer auditable data-trust evidence. **$0 runtime** —
in-process library, no new cloud service.

## What to build — the hybrid (research §3–5, PoC-verified against dlt 1.28.2)

### Option A — LOAD (Data Engineer). PoC-proven, low risk.
- `oss2` read (proven) → Arrow → **dlt pipeline → DuckDB destination**, with
  `write_disposition="merge"`, `primary_key="loan_id"`, `schema_contract` (freeze).
- Refactor `lakehouse.py`'s load step to materialize via dlt; the Data Engineer's quality tools
  query the dlt-loaded dataset. Point `pipelines_dir` at FC's writable `/tmp`.
- **Data Engineer GAINS (research §8):** cite dlt load-freshness / rows-loaded / schema-contract
  pass-fail / schema-drift from `_dlt_loads` as **auditable debate evidence**.

### Option B — WRITE (Data Analyst → medallion). Needs `ossfs` + a live-cred smoke test.
- **dlt filesystem destination** (`bucket_url="oss://waspada-prod-staging|mart"`, parquet) writes
  the Analyst's derived features/aggregates to Staging/Mart.
- **Activates the medallion write-policy buckets** the IaC grants PutObject to but code never uses
  (closes the WA-070 gap).
- Adds `ossfs` + hand-wired OSS `endpoint`/`key`/`secret` (`oss` is NOT in dlt's typed creds).

## Acceptance
- Data Engineer loads the book via dlt (contract + merge + lineage); quality tools query it.
- `_dlt_loads` lineage (load_id, rows, schema version) surfaced in the audit + citable by the Engineer.
- (Option B) Analyst features written to OSS Staging/Mart as parquet; verified round-trip.
- **Offline / tests stay green:** dlt runs in-process on the synthetic snapshot with **no cloud
  creds**; the mock / train-per-run paths byte-for-byte unchanged.
- Tests: dlt load round-trip (merge-dedup + contract-reject), lineage-in-audit, Option-B write (mocked fs).

## Out of scope
- **dlt's MCP server as a RUNTIME component** — researched + rejected (research §7): redundant with
  WASPADA's own purpose-built MCP, a generic `execute_sql_query`/admin surface, heavy deps. dlt stays
  **under** our MCP, not beside it. (dlt's MCP is fine as a **dev-time** accelerator for *building* this.)
- **httpfs / `read_parquet('s3://…')` read pushdown** — that's WA-047's remaining read-path scope; a
  dlt load is not pushdown.
- **`requirements.txt` dlt line** — already present (WA-081: stays). Do not touch it.
