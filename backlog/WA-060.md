# WA-060 · RDS MySQL DuckDB analytical read-only instance + OSS Parquet external tables

## The gap

WASPADA's lakehouse is OSS + embedded DuckDB (client-side, in-process). The
Alibaba Cloud **RDS MySQL DuckDB analytical read-only instance** is a managed
service that puts the DuckDB engine *inside* RDS MySQL as a storage engine —
column-store, vectorized, 100–200× faster than InnoDB for analytical queries,
binlog-replicated from the primary in real time.

This is the rubric's "sophisticated Alibaba Cloud service usage" play: instead
of just running DuckDB locally, we run it *as a managed Alibaba service*.

**Two RDS instances, not one:**

```
┌──────────────────────┐     binlog replication     ┌──────────────────────────┐
│  RDS MySQL (Primary) │ ──────────────────────►   │  RDS MySQL DuckDB        │
│  mysql.n2.medium.1   │                            │  (Read-Only Analytical)  │
│  - InnoDB            │                            │  - DuckDB engine         │
│  - Auth (WA-028)     │                            │  - Column store, OLAP    │
│  - OLTP              │                            │  - 100–200× faster reads │
│  - Running ✅        │                            │  - NOT provisioned ❌    │
└──────────────────────┘                            └──────────────────────────┘
         ▲                                                    │
         │                                              reads Parquet from OSS
         │                                              via S3-compatible API
         │                                                    │
┌────────┴────────────────────────────────────────────────────┘
│  OSS Buckets (lakehouse: raw/ staging/ mart/)
│  waspada-prod-raw, waspada-prod-staging, waspada-prod-mart
└──────────────────────────────────────────────────────────────┘
```

## Architecture decision (owner, 2026-07-18)

**Primary RDS MySQL stays as-is** (auth + OLTP). A **DuckDB analytical
read-only instance** attaches to it for OLAP. The DuckDB instance reads
Parquet from OSS via the S3-compatible endpoint for lakehouse analytics.

### 2. OSS S3-compatible external tables — DuckDB in RDS

The DuckDB engine in RDS can read Parquet from OSS via the S3-compatible
API. The existing `_oss_s3_endpoint()` in `waspada/data/lakehouse.py:67-83`
already maps OSS credentials to S3 settings.

For the **RDS DuckDB instance**, the pattern is:
```sql
-- Inside the DuckDB RDS instance (via MySQL client)
-- OSS S3-compatible endpoint: https://<bucket>.oss-ap-southeast-1.aliyuncs.com
-- Or internal: https://<bucket>.oss-ap-southeast-1-internal.aliyuncs.com

-- The DuckDB engine supports httpfs/S3 natively
SELECT * FROM read_parquet('s3://waspada-prod-raw/lane=collections/as_of=2026-07-18/loans.parquet');
```

**Key config (via SQL SET commands in the DuckDB RDS instance):**
```sql
SET s3_endpoint='oss-ap-southeast-1.aliyuncs.com';
SET s3_access_key_id='<OSS_ACCESS_KEY_ID>';
SET s3_secret_access_key='<OSS_ACCESS_KEY_SECRET>';
SET s3_region='ap-southeast-1';
-- For VPC-internal access (no internet egress):
-- SET s3_endpoint='oss-ap-southeast-1-internal.aliyuncs.com';
```

**External table pattern (persistent, reusable):**
```sql
-- Create a view that reads from OSS
CREATE VIEW raw_loans AS
SELECT * FROM read_parquet('s3://waspada-prod-raw/lane=collections/as_of=*/loans.parquet');

-- Query it like a normal table
SELECT loan_id, p_default, score_band, final_band
FROM raw_loans
WHERE as_of = '2026-07-18'
ORDER BY p_default DESC
LIMIT 20;
```

### 3. App integration — `waspada/data/lakehouse.py`

Extend the existing `load_to_duckdb()` to support a **remote DuckDB RDS
endpoint** as an alternative to the local embedded DuckDB:

```python
# New env var
DUCKDB_RDS_ENDPOINT = os.environ.get("DUCKDB_RDS_ENDPOINT", "")  # e.g. rm-xxx.mysql.singapore.rds.aliyuncs.com
DUCKDB_RDS_PORT = int(os.environ.get("DUCKDB_RDS_PORT", "3306"))

def get_analytics_connection():
    """Return a connection to the DuckDB RDS analytical instance, or local DuckDB."""
    if DUCKDB_RDS_ENDPOINT:
        import pymysql
        return pymysql.connect(
            host=DUCKDB_RDS_ENDPOINT,
            port=DUCKDB_RDS_PORT,
            user="waspada",
            password=os.environ["RDS_PASSWORD"],
            database="waspada",
        )
    # Fall back to local embedded DuckDB
    return duckdb.connect(":memory:")
```

The app queries the DuckDB RDS instance via the **MySQL protocol** (same as
the primary). The DuckDB engine handles the analytical SQL; the MySQL layer
handles auth and connection management.

### 4. Data sync — primary → DuckDB analytical

Once the DuckDB read-only instance is attached:
- **Historical data:** Alibaba syncs automatically on creation (converts all
  InnoDB tables to DuckDB format)
- **Incremental data:** binlog replication, real-time, no external tools
- **OSS data:** Not synced via binlog — the DuckDB instance reads Parquet
  from OSS directly via S3-compatible API (external tables, not replicated)

**Schema note:** DuckDB instances auto-convert InnoDB tables. Foreign keys
are dropped. Tables without a primary key are not converted (stay InnoDB).
Our auth tables (users, sessions) have PKs — they'll convert fine.
