# WA-027 · Alibaba Cloud IaC with OpenTofu (replaces manual WA-018 deploy)

## Goal

Provision WASPADA's Alibaba Cloud infrastructure as code with OpenTofu (the
`alicloud` provider), so the whole cloud footprint is declared, reviewable,
and reproducible — and the config doubles as submission proof of Alibaba
Cloud usage. Region: ap-southeast-1 (Singapore / international, for the free
tier). Bimo authors; the owner runs `tofu apply` (touches real spend → owner
sign-off gate).

## Files / interfaces

- `deploy/iac/main.tf` · `variables.tf` · `outputs.tf` · `versions.tf`
  declaring:
  - `alicloud` provider (Singapore region), AccessKey via `TF_VAR_*` env /
    gitignored `*.tfvars` — NEVER committed.
  - OSS bucket for the loan-portfolio Parquet (the WA-018/WA-023 data store).
  - Container Registry (ACR) repo for the FC image (or reference an existing).
  - Function Compute 3.0 custom-container function (CAPort 8080, points at the
    ACR image) + HTTP trigger → public `*.fcapp.run` URL.
  - Simple Log Service project + logstore (the WA-023 audit stream).
  - RAM role granting FC read on OSS + write on SLS.
  - Outputs: the public FC URL, OSS endpoint, SLS project/logstore names.
- `deploy/iac/README.md` — runbook: `tofu init` / `plan` / `apply` / `destroy`,
  plus the console/Serverless-Devs fallback steps.
- `.gitignore` additions: `*.tfstate*`, `*.tfvars`, `.terraform/`.

## Acceptance

- `tofu init && tofu plan` succeed cleanly with the `alicloud` provider.
- `tofu apply` provisions all resources above in one pass; `tofu destroy`
  tears them down.
- The output FC URL serves `/api/health` → `{"status":"ok"}`.
- No secret is committed (state, tfvars, AccessKey all gitignored — verify
  with `git status` / `git check-ignore`).
- README runbook lets a fresh checkout reproduce the stack from zero.

## Out of scope

- Remote/OSS state backend (local state is fine for the hackathon).
- CI/CD automation of `apply` (owner runs it manually — spend gate).
- Multi-environment / staging. The `loans.parquet` upload may stay a one-line
  script or an optional `alicloud_oss_bucket_object` — not required for done.

## Notes

- This ticket subsumes WA-018's manual deploy runbook into declared IaC.
  WA-018 stays as the "owner runs the deploy" gate; WA-027 is the code.
- The Dockerfile already targets CAPort 8080 — do NOT change it.
- FC sandbox + stdio MCP is unverified (HACKATHON.md risk) — the IaC config
  should not block on it; MCP runs locally/CI while FC serves the app.
