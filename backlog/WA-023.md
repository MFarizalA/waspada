# WA-023 · SLS audit-log stream (~2h)

## Goal

Ship every `Step`/`Handoff`/`DisputeRound` to Alibaba Simple Log Service as
structured logs — SQL-queryable in the SLS console. This is the "show me the
audit trail" answer for a regulated lender, and the fourth Alibaba Cloud service
(OSS + Function Compute + Qwen Cloud + SLS) — strengthens the Alibaba-native
story for the rubric.

## Files / interfaces

- `waspada/audit/sls.py` (NEW) — thin wrapper on `aliyun-log-python-sdk`.
  Fields: `run_id, agent, action, model, tokens, latency, resolution`.
- Wired onto the orchestrator's existing step log (one emit per step).
- Fail-safe: SLS unavailable → log locally, NEVER block the pipeline.

## Acceptance

- A pipeline run emits structured logs to SLS (visible in the console).
- SLS down/unreachable → pipeline completes; logs fall back to local file.
- Fields match the spec (`run_id/agent/action/model/tokens/latency/resolution`).
- All existing tests green (SLS call must be mockable/skippable in CI).

## Out of scope

- A query UI — the SLS console is the query surface (screenshot for the demo).
- Streaming logs (emit-on-step is sufficient).

## Depends on

- WA-014 (disputes/rounds to log).
- Owner's Alibaba AccessKey (same pair as OSS) — blocked until creds land.

## Notes for the worker

- ~2h scope — keep it thin. Free tier covers demo volume.
- The fail-safe is non-negotiable: an SLS outage must never break a collections run.
