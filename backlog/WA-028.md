# WA-028 · Authentication (login, register, forgot password)

## Goal

Add user authentication to the WASPADA dashboard: register, login, forgot
password. The analyst signs in to see the work-list and approve disputes;
the gate actions (resolve_risk_dispute, publish_work_list) become
attributable to a named human, not an anonymous auto-approve.

## Scope discipline (read before building)

This is a hackathon with ~30h left. The auth must be the SMALLEST correct
thing that works for the demo — not a production identity system.

- JWT-based session (no OAuth/OIDC complexity).
- Passwords hashed (bcrypt or argon2 — never plaintext).
- Forgot password: generate a reset token, store it, email it (or log it in
  dev — no SMTP dependency required for the demo). The mechanism exists and
  is testable; actual email delivery is a stretch.
- Store users in **ApsaraDB RDS PostgreSQL** (Alibaba Cloud managed DB —
  adds a 5th Alibaba service to the submission). Connect via `DATABASE_URL`
  env var (standard PostgreSQL connection string). Use SQLAlchemy Core or
  raw psycopg2 — no heavy ORM needed for 4 routes. Create the `users` table
  on startup if it doesn't exist (auto-migrate). For local dev without RDS,
  fall back to a local PostgreSQL or SQLite via the same SQLAlchemy engine
  (the connection string determines the driver — keep it abstract).

## RDS schema

```sql
CREATE TABLE IF NOT EXISTS users (
    id          SERIAL PRIMARY KEY,
    email       TEXT UNIQUE NOT NULL,
    password    TEXT NOT NULL,  -- bcrypt hash, never plaintext
    created_at  TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE IF NOT EXISTS reset_tokens (
    token       TEXT PRIMARY KEY,
    email       TEXT NOT NULL REFERENCES users(email),
    expires_at  TIMESTAMPTZ NOT NULL,
    used        BOOLEAN DEFAULT FALSE
);
```

## Files / interfaces

- `api/auth.py` (NEW) — FastAPI auth routes:
  - `POST /api/auth/register` — email + password → create user, return JWT
  - `POST /api/auth/login` — email + password → verify, return JWT
  - `POST /api/auth/forgot-password` — email → generate reset token, store
  - `POST /api/auth/reset-password` — token + new password → update, invalidate token
- `api/db.py` (NEW) — thin SQLAlchemy/raw-psycopg2 connection helper.
  Reads `DATABASE_URL` env var; creates tables on startup.
- `api/main.py` — wire the auth router; protect /api/run behind a JWT check
  (the dashboard requires login to run the pipeline)
- Frontend (Kirana, separate task): login/register/forgot-password forms;
  store JWT in localStorage; redirect to login when 401

## Acceptance

- Register → login → authenticated /api/run works end-to-end
- Wrong password → 401
- Forgot password → reset token generated → reset-password → new password works
- Passwords are hashed (never stored plaintext)
- All existing tests stay green
- New tests cover: register, login (success + failure), reset flow

## Out of scope

- OAuth / SSO (Google, GitHub, etc.)
- Email delivery (reset token can be logged in dev)
- Role-based access control (single role: analyst)
- Session expiry / refresh tokens (JWT with reasonable TTL is fine)
- Production user store (SQLite or in-memory for the demo)

## Notes for the worker

- Use python-jose or PyJWT for JWT; passlib[bcrypt] for hashing.
- The demo can seed a default analyst user on startup (so the judge doesn't
  have to register before clicking around).
- Keep the auth boundary simple: /api/run and /api/auth/reset-password need
  auth; /api/auth/login, /api/auth/register, /api/health stay public.
