# WA-035 · `features/origination.py` — application-time features + label

## Goal

The origination feature recipe, parallel to `features/collections.py` (which is
collections top-to-bottom: `is_default`/`build_label` read `current_status`
(`collections.py:72-125`), `build_features` computes `payment_ratio`/
`outstanding_ratio` from payment history (`:186-187`) — none of it applies to an
application).

## Files / interfaces

- `waspada/features/origination.py` —
  `build_features(raw: RawApplications, as_of) -> ApplicationFeatureFrame` +
  `build_label(...)`. Candidate features (all application-time, no leakage):
  `amount`, `term`, `requested_rate`, `grade`, `annual_income`, `dti`,
  `loan_to_income = amount/annual_income`, `purpose`, `region`,
  `employment_length`. Label: funded-then-defaulted (WA-034 honesty note).
- Same discipline as collections: contract-validated output, `assert_no_nulls`,
  deterministic (the Data Analyst LLM explores; it never computes the frame —
  same rule as WA-030).
- **Leakage guard is the acceptance centerpiece**: nothing outcome-derived
  (`funded_default`, any post-booking field) may enter the feature matrix —
  mirror `LEAKAGE_EXCLUDED` + its test (`model/risk.py:81-86`,
  `tests/qa/test_leakage.py`).

## Acceptance

- Offline synthetic `RawApplications` → valid `ApplicationFeatureFrame`, no nulls.
- Leakage test: label not derivable from features.
- Mirrors `tests/test_features.py` structure.
