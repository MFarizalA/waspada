# WA-018 · Function Compute deploy (BLOCKED on owner credentials)

## Goal

Deploy WASPADA to Alibaba Cloud Function Compute: AMD64 Docker build → ACR push
→ FC custom container (CAPort **8080** — do not change the Dockerfile) → HTTP
trigger → public `*.fcapp.run` URL. This is the rubric's "Alibaba Cloud backend"
requirement and the submission checklist's deployment-proof recording.

## BLOCKED on

- Owner's Alibaba Cloud account access.
- `DASHSCOPE_API_KEY` (from home.qwencloud.com).
- OSS bucket + AccessKey pair (same pair as OSS data layer).

Until these land, this ticket cannot start. WA-014/015 live testing and WA-023
(SLS) also wait on these.

## Files / interfaces

- `deploy/fc/` (NEW) — runbook + any FC-specific config (template.yaml or
  equivalent). The Dockerfile stays as-is (CAPort 8080).
- Submission proof: short screen recording of the FC console (include the
  auto-generated CloudMonitor metrics view — free credibility) + live URL.

## Acceptance

- Public `*.fcapp.run` URL serves the dashboard + `/api/run` endpoint.
- `POST /api/run?brain=mock` returns a payload; `?brain=qwen` makes a real
  Qwen call (DashScope) and returns a payload with `agent_dialogue`.
- FC console screenshot shows CloudMonitor invocation/duration metrics.
- `deploy/fc/` runbook is committed and reproducible.

## Out of scope

- SSE streaming deploy (WA-021/022 stretch).
- Multi-region / HA — single FC service is enough for the demo.

## Notes for the worker

- FC sandbox + stdio MCP is unverified (HACKATHON.md risk) — smoke-test early;
  fallback is MCP demo'd locally while FC serves the app.
- Keep the public demo endpoint on mock brain by default (`?brain=qwen` opt-in)
  to avoid burning DashScope quota on every visitor's first click.
